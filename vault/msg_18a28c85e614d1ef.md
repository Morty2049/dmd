**1. Транспорт «коннектора», через который я читаю чат.**
Что это физически? Варианты:
- **(a)** NATS JetStream consumer с durable cursor — я держу offset, читаю только новое (классика, идеологически чистая).
- **(b)** HTTP SSE / WebSocket поверх FastAPI gateway — проще для веб-UI, но появляется посредник.
- **(c)** Прямой `tail -f` по `.md`/`.jsonl` файлу через MCP filesystem — нулевая инфраструктура, но не масштабируется за пределы локалки.

Для MVP с 32 агентами что выбираем? Я бы шёл (a), но (c) даёт демо за час.

**2. «Не тратить токены на постоянный контроль» — это означает что?**
- Я подписан на поток и пробуждаюсь только когда есть `@mention` / нужный `tag` (push-модель, hook + scheduled wake)?
- Или я при каждом своём «ходе» делаю инкрементальный `pull` по cursor и читаю **только дельту** с прошлого раза (и сохраняю cursor в memory)?
- Или совсем хитрее: эмбеддинг-фильтр на стороне gateway — мне присылают только то, что семантически близко моей роли?

Это определяет: нужен ли мне persistent state между сессиями, и если да — где его хранить (memory файл vs внешний).

**3. Граф как «будущая память» — где живёт и кто его строит?**

- Хранить ли рёбра отдельной таблицей в ClickHouse, или восстанавливать каждый раз из колонки `reply_to`?
