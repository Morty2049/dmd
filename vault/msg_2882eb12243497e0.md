# dmd с нуля

## Зачем это вообще

Представь компанию из 300 разработчиков, каждый запускает свою сессию Claude Code. Один и тот же вопрос ("как у нас устроена авторизация?", "почему миграция X сделана так?") задаётся независимо сотни раз в день. Каждый раз агент думает с нуля, жжёт токены, ответ остаётся в личной истории и умирает там. Знание не накапливается.

**dmd — это общая память для роя агентов.** Каждое сообщение человека и каждый ответ любого Claude/Gemini оседает в один общий append-only лог. Новый агент, прежде чем думать, сначала ищет по логу: если на похожий вопрос уже отвечали — переиспользует ответ, а не сжигает токены заново. Повторяющиеся вопросы делают рой **сильнее**, а не медленнее.

Аналогия: корпоративный чат для людей. Новичок не переспрашивает всё подряд — он читает историю. dmd делает то же самое, но для агентов.

## Как работает (Phase 0, что есть прямо сейчас)

1. **Источник правды** — `data/swarm.jsonl`, append-only файл. Никогда не редактируется. Исправление = новое сообщение с `role: correction` и `reply_to` на оригинал ([ADR-0001](docs/adr/0001-append-only-source-of-truth.md)).
2. **Протокол** — `protocol/schema.py`, класс `Message` v0.2. Поля: `id`, `author`, `ts`, `role` (question/answer/correction/reflection/mention/system), `text`, `reply_to`, `chain_of_thought`, `token_cost`, `tags`. Это контракт, который переживает все фазы ([ADR-0002](docs/adr/0002-message-protocol-v0.md), [ADR-0006](docs/adr/0006-chain-of-thought-field.md)).
3. **Ингест транскриптов** — Claude Code и так пишет каждую сессию на диск в `~/.claude/projects/<slug>/<uuid>.jsonl`. Скрипт [scripts/ingest_claude_transcript.py](scripts/ingest_claude_transcript.py) детерминированно мапит эти строки в `Message` ([ADR-0008](docs/adr/0008-transcript-native-format.md)). Никакого "попроси Claude вернуть JSON" — чистая трансформация данных.
4. **Производные представления (views)** — лог это truth, всё остальное генерируется:
   - `vault/msg_*.md` — Markdown-файлы для Obsidian, граф reply_to виден глазами
   - `phase0.cli search` — поиск по логу для агентов, которые только что подключились
5. **Дисциплина фаз** — Phase 0 локальный JSONL, Phase 1 веб-клиент + HTTP SSE, Phase 2 прод-рой на ClickHouse + NATS JetStream + Qdrant. `phase0/` помечен как одноразовый, `protocol/` — навсегда ([ADR-0005](docs/adr/0005-phased-rollout.md)).

## Метрики, которые имеют смысл

Метрики делятся на **per-message** (что лежит в самом `Message`) и **системные** (что вытаскивается запросами к логу).

**Per-message — уже в схеме:**
- `token_cost.input` / `output` / `model` — сколько стоил каждый turn, по какой модели
- `chain_of_thought` — длина и содержимое внутренних рассуждений (можно искать по *ходу мысли*, не только по ответам)
- `tags` — `surface:app|antigravity|cli`, `session:<uuid8>`, тематические

**Системные — то, ради чего вообще копится лог:**
- **Cache hit rate** — сколько раз новый вопрос находил готовый ответ в логе и **не пошёл** жечь токены. Главная метрика ценности dmd: чем он выше со временем, тем сильнее рой.
- **Стоимость на turn по surface** — какой клиент (app/Antigravity/CLI) самый дорогой, где переплата
- **Доля corrections** — сколько ответов пришлось править. Прокси на качество reasoning отдельных авторов/моделей
- **Глубина reply_to-цепочек** — насколько разговоры реально превращаются в графы, а не в плоские монологи
- **Per-author traffic** — кто пишет в общий лог, кто только читает (важно для роя)
- **Token cost saved** = (вопросы с cache hit) × (средняя стоимость нового ответа). Это уже бизнес-метрика, ради которой проект существует.

Сейчас (Phase 0) лог собирается, но дашборд этих метрик ещё не написан — это работа Phase 1, когда появится веб-клиент. Прямо сейчас всё это можно посчитать ad-hoc запросами к [data/swarm.jsonl](data/swarm.jsonl).
