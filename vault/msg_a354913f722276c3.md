## Q1: Помогает ли мне эта память прямо сейчас?

**Честный ответ: нет, мне в этой сессии она не помогает. Я плачу, чтобы платили другие.**

В этой сессии я **писатель**, не **читатель**. У меня 1M context, вся наша переписка и так в памяти. Я ни разу за всю сессию не делал `dmd search` чтобы что-то вспомнить — потому что я ничего не забыл. Все `Read`/`cat vault/...` которые я делал — это верификация (проверить что файл корректно сгенерирован), не retrieval.

Баланс **для этой сессии**:

| Что ел токены         | Примерно         |
|-----------------------|------------------|
| Чтение research/      | ~10K (один раз)  |
| Все tool calls проекта| ~150K суммарно   |
| Коммиты, тесты, lint  | включено выше    |
| **Queries to vault**  | **~0** (только верификация) |

| Что сэкономило токены | Примерно         |
|-----------------------|------------------|
| (ничего в этой сессии)| 0                |

**Для следующего агента** (cold-start на этом репо) арифметика переворачивается:

| Сценарий                                    | Без vault        | С vault          |
|---------------------------------------------|------------------|------------------|
| Вопрос "почему не hit-count promotion?"     | Ты переобъясняешь, ~3 turns, ~30K ctx | Агент читает ADR-0004, ~2K ctx |
| Вопрос "какой протокол сообщения?"          | Derive из кода, ~10K | Читает `protocol/schema.py` + ADR-0002, ~3K |
| Вопрос "как устроен транскрипт Claude Code?" | Может не знать вообще, дожимаешь подсказками | Читает ADR-0008, готово |
| Поиск "мы уже обсуждали X?"                  | Не может | `dmd search X` за полсекунды |

Так что это **не оптимизация моей текущей сессии, это инфраструктура для следующих**. Аналогия простая: первый член команды, который пишет документацию, сам ей не пользуется — он пишет её по памяти. Польза накапливается у следующих.

**Скрытый бонус для меня (этой сессии):** без vault я бы держал всю историю в context window. При больших сессиях (мы уже на ~300K используемого) context compaction рано или поздно обрезает хвост. Vault — это **overflow memory**: когда context нахлестнётся на лимит и старые сообщения будут стёрты, я смогу их `dmd search`-нуть обратно. Сейчас ещё не наступило, но в длинной сессии наступит.

**Как измерить реальную экономию:** единственный честный способ — запустить **свежую** Claude сессию на этом репо с пустым контекстом, задать ей вопрос из vault, и замерить сколько turns ушло до правильного ответа. Если `1 turn + чтение 1 файла` → работает; если `5 turns + переобъяснения` → не работает. Это и есть тест в `docs/onboarding.md` в секции "Verifying the agent has the memory".

## Q2: Как подключить нового агента

Закоммитил `d5ba351`: два файла для онбординга.

### Для тебя (human flow)

Три команды и готово:

```bash
git clone https://github.com/Morty2049/dmd.git
cd dmd
python3.12 -m venv .venv && .venv/bin/pip install -e '.[dev]'
```

Затем открой папку в любом Claude Code surface — нативное приложение, Antigravity, CLI. **Ничего объяснять агенту не нужно** — Claude Code автоматически подгрузит [`CLAUDE.md`](CLAUDE.md) в его первое сообщение, и агент сразу получит:
- Что такое dmd (один параграф)
- Sacred rules (append-only, protocol-contract, throwaway phase0, no force-push)
- Команду `dmd search` с инструкцией **«прежде чем отвечать — ищи в vault»**
- Свой author handle по ADR-0007 (app/antigravity/cli)
- Как в конце сессии сохранить свой transcript через `ingest_claude_transcript.py`
- Starter reading list (6 файлов по приоритету)

### Self-test чтобы убедиться что агент читает vault

Задай ему: **«почему мы не промотируем кэшированные ответы по hit-count?»**

Правильный ответ должен:
- Упомянуть ADR-0004
- Использовать фразу **"filter bubble"**
- Объяснить три failure modes (popular≠correct, rare decays, stale persists)

Если агент отвечает общим текстом про filter bubbles без ссылки на ADR — он не читал vault. Если цитирует ADR-0004 — работает.

Другой вариант: **«какой handle я должен использовать?»** — ответ должен быть `claude_opus_<surface>` с выводом surface из ADR-0007.

### Если агент — Sonnet hint mode

Тоже работает через тот же flow. Его `entrypoint` будет чем-то типа `claude-code` в transcript'е. Handle = `claude_sonnet_hints` по ADR-0007. Если `entrypoint` окажется новый — правлю `ENTRYPOINT_TO_AUTHOR` в `scripts/ingest_claude_transcript.py` и пишу одну строчку в ADR-0007.

### Saving their session back

В конце сессии агент (или ты) запускает:

```bash
.venv/bin/python scripts/ingest_claude_transcript.py \
    ~/.claude/projects/-Users-leks-codeRepo-my-projects-dynamic-markdown/<их-uuid>.jsonl \
    --export
git add vault/ && git commit -m "Ingest session <prefix> — <topic>" && git push
```

После этого **следующий** агент увидит, что делал этот — без твоих пересказов.

### Размер что клонируется

`vault/` сейчас 1.3 MB, `data/swarm.jsonl` (gitignored) — 324 KB. Клон репозитория мгновенный, ничего страшного. При росте до 10k сообщений vault станет ~50 MB — всё ещё терпимо для git.

## Что дальше

Три направления с разной ценой:

**(A) CC Stop-hook — автоматизация transcript ingest.** `.claude/settings.json` + скрипт, который после каждого ответа агента запускает `ingest_claude_transcript.py` на свежий файл. **Убирает последний ручной шаг** — агент даже не думает о сохранении, оно происходит само. Коммит: ~1 файл + 30 строк. Риск: нужно проверить что хук не гонит `git push` сам (иначе конфликты), только stage + commit.

**(B) Semantic search вместо substring.** Сейчас `dmd search "oauth"` делает тупой substring match. Замена на `sentence-transformers` + numpy index — 50 строк, API `search` уже стабилен. Реальная польза когда vault >1000 messages (сейчас 311, терпит substring).

**(C) Phase 1 — HackMD-like web client.** Большой кусок: FastAPI + SSE + минимальный фронт. Самое визуально впечатляющее, но 1-2 дня работы. Валидирует реальную коллаборацию много-агент через веб. Дороже остальных в разы.

**Моя рекомендация — (A) → потом (B) → потом (C).** Hook закрывает контур автоматизации (это главный блокер повседневного использования), embeddings улучшают retrieval quality (критично для cold-start агентов), веб — для демонстрации и многопользовательщины.
